## Padrão escolhido: Factory Method

### Intenção (definição GoF)

> "Definir uma interface para criar um objeto, mas deixar as subclasses decidirem qual classe instanciar. O **Factory Method** permite que uma classe delegue a instanciação para subclasses." (GAMMA et al., 1994)

É um padrão **criacional** que substitui chamadas diretas a `new ConcreteProduct()` no código cliente por chamadas a um método polimórfico (*factory method*). Com isso, a decisão de *qual* subtipo instanciar fica isolada em uma hierarquia de *Creators*, favorecendo o princípio Aberto/Fechado (OCP).

### Estrutura

- **Product** (interface/classe abstrata): define a interface dos objetos produzidos. No Battle Class: `Inimigo`, `Questao`, `Torre`.
- **ConcreteProduct**: implementa `Product`. Ex.: `InimigoMatematica`, `InimigoHistoria`, `TorreEspecial`.
- **Creator** (abstrato): declara `factoryMethod(): Product` e contém a lógica de negócio que *usa* o produto pela interface (ex.: `spawn(onda)` escalando HP).
- **ConcreteCreator**: sobrescreve `factoryMethod()` retornando um `ConcreteProduct` específico.

### Quando usar / quando não usar

- **Use quando** a classe não pode antecipar quais objetos concretos deve criar; quando se deseja delegar a decisão de criação a subclasses; quando há uma família crescente de variantes do mesmo produto e é desejável adicionar novas variantes sem alterar o código cliente.
- **Não use quando** há apenas um ou dois tipos estáveis (um `new` direto basta), ou quando é preciso criar *famílias* de produtos relacionados — nesse caso prefira *Abstract Factory*.

### Senso crítico — diferenças relevantes

- **vs. Abstract Factory:** Factory Method usa *herança* e cria **um** produto; Abstract Factory usa *composição* e cria *famílias* de produtos relacionados (frequentemente implementado com vários Factory Methods internamente).
- **vs. Simple Factory** (*idiom*, não-GoF): um único método com `switch`/`if` retornando subtipos — viola OCP. O Factory Method substitui o `switch` por polimorfismo via subclasses.

### Aplicação ao Battle Class

- **(a) Inimigos por matéria/dificuldade.** Cada onda do *Tower Defense* é temática e ligada a uma matéria. Um `InimigoCreator` abstrato com subclasses `InimigoMatematicaCreator`, `InimigoHistoriaCreator`, `InimigoBiologiaCreator` produz inimigos com HP, velocidade e *sprite* calibrados ao tema, permitindo novas matérias sem tocar no *loop* de *spawn*.
- **(b) Questões por banca.** `QuestaoCreator` abstrato com `QuestaoEnemCreator`, `QuestaoFuvestCreator`, `QuestaoUnbCreator` — cada banca tem formato/*parser*/normalização próprios. O *front-end* consome um `Questao` uniforme.
- **(c) Torres e *loot*.** `TorreCreator`/`TorreEspecialCreator` (custo e *upgrade* por nível) e a roleta de matérias via `RoletaCreator`.
- **Ponto único de resolução.** O [`CreatorRegistry`](https://github.com/UnBArqDsw2026-1-Turma02/2026.01-T02_G6_Battle_Class_Entrega_03/blob/main/src/server/factories/index.ts) mapeia `Materia → InimigoCreator` e `Banca → QuestaoCreator`, eliminando `if/else` sobre tipos concretos nos serviços de aplicação (`VestibularServiceFacade`, `TowerDefenseService`).
