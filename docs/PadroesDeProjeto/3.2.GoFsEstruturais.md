## Padrão escolhido: Facade

### Intenção (definição GoF)

> "Fornecer uma interface unificada para um conjunto de interfaces em um subsistema. O **Facade** define uma interface de nível mais alto que torna o subsistema mais fácil de usar." (GAMMA et al., 1994)

É um padrão **estrutural**: reduz acoplamento entre clientes e um subsistema complexo, oferecendo um *ponto de entrada único*. Apoia o *Princípio do Menor Conhecimento* (Lei de Demeter): o cliente conversa com poucas classes.

### Estrutura

- **Client**: chama apenas a Facade; não conhece o subsistema.
- **Facade**: classe com métodos de alto nível; mantém referências às classes do subsistema e delega chamadas a elas.
- **Subsystem classes**: implementam a funcionalidade real; *não* conhecem a Facade e podem ser usadas diretamente em casos avançados.

### Quando usar / quando não usar

- **Use** quando há um subsistema complexo com muitas classes interdependentes; quando se deseja desacoplar clientes dos detalhes internos; quando é preciso oferecer um *ponto de entrada* simplificado (ex.: API pública, casos de uso); quando há camadas e quer-se proteger as superiores.
- **Não use** se o subsistema já é simples (gera burocracia inútil); se você precisa traduzir interfaces incompatíveis (use *Adapter*); se precisa coordenar comunicação bidirecional entre objetos (use *Mediator*).
- **Antipadrão a evitar:** permitir que a Facade vire um *God Object* acumulando regra de negócio — é sinal de que ela deveria ser quebrada em fachadas menores ou em casos de uso. Por isso o Battle Class optou por **quatro Facades** especializadas em vez de uma só.

### Senso crítico — diferenças relevantes

- **vs. Adapter:** Adapter *converte* uma interface existente em outra esperada (compatibilidade); Facade *cria* uma interface nova e simplificada sobre várias.
- **vs. Mediator:** Mediator centraliza comunicação entre colegas que se conhecem por ele (bidirecional); Facade é unidirecional (cliente → subsistema), e o subsistema ignora a fachada.
- **vs. Proxy:** Proxy tem a *mesma* interface do objeto real e controla acesso (lazy, remoto, cache, segurança); Facade tem interface *diferente* e agrega múltiplos objetos.

### Aplicação ao Battle Class

- **`PartidaFacade`.** Ao terminar uma rodada, unifica `QuizService` (corrigir respostas), `EconomiaService` (creditar moedas), `TowerDefenseService` (liberar ondas/torres) e `PartidaRepository` (Supabase). O *controller* Express ou um *hook* React chama apenas `partidaFacade.finalizarRodada(userId, respostas)`.
- **`VestibularServiceFacade`.** Agrega múltiplas fontes de questões (ENEM, FUVEST, UnB) atrás de `obterQuestoes(filtros)`, escondendo as bancas, o *parsing* e o *cache* — resolvendo o Creator certo via `CreatorRegistry`.
- **`AuthFacade`.** Encapsula Supabase Auth + perfil + emissão/validação de token (`registrar`, `login`, `validarToken`).
- **`RankingFacade`.** Agrega `RankingRepository` + `CarteiraRepository` para o *leaderboard* (`top`, `posicaoDoUsuario`).
- **`ConfigService`.** Subsistema **compartilhado** (lê *flags* de ambiente) injetado na mesma instância em mais de uma Facade, sem que uma conheça a outra — evita "Facade chamando Facade".
